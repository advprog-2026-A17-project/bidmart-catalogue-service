# Software Design

## Scope

Product listing lifecycle, marketplace search/filter, seller listing management, category resolution, bid-state projection, and listing expiry reconciliation.

## Module Design

`ListingServiceImpl` is a compatibility facade over command/query handlers, listing validation chain, listing state objects, and expiry strategies.

## SOLID Mapping

| Principle | Implementation |
|-----------|----------------|
| SRP | Controllers/routers remain thin while use-case handlers, validators, repositories, adapters, and schedulers own focused responsibilities. |
| OCP | New flows can be added through handlers/strategies/factories without modifying facade contracts. |
| LSP | Strategy, state, adapter, and client implementations are substitutable behind their interfaces/traits. |
| ISP | Interfaces are small and workflow-specific rather than one broad service abstraction. |
| DIP | Application code depends on repositories/client abstractions and composition roots wire concrete infrastructure. |

## Design Patterns

Facade, Command handlers, Chain of Responsibility validation, State for `ListingStatus`, Strategy/Factory for expiry outcome, Adapter for gRPC/external integration.

## Compatibility Rule

Public endpoint paths, DTO shapes, routing keys, and existing constructor/service methods are kept compatible; internal refactors are implementation details.
