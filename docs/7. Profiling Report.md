Dalam sesi ini, Saya berhasil mengidentifikasi, memprofiling, dan menyelesaikan bottleneck (hambatan) performa yang sangat kritikal pada layanan Catalogue Service. Masalah ini sebelumnya menyebabkan waktu muat (loading time) yang lambat pada halaman Seller Studio di Frontend, terutama ketika basis data mulai membesar.

Berdasarkan pengujian langsung (dengan menyuntikkan 10.000 data fiktif ke dalam basis data), berikut adalah perbandingan metrik performa di Postman:





Penjelasan: 
Penurunan waktu respons dari 150 milidetik menjadi hanya 30 milidetik terjadi karena sebuah pergeseran paradigma arsitektur: kita mengubah lokasi di mana proses penyaringan (filtering) data dilakukan.

Paradigma Lama (Client-Side Filtering)
Sebelumnya, sistem Anda mengambil pendekatan "Tarik Semua, Buang Belakangan".

Frontend meminta data menggunakan endpoint GET /api/v1/catalogue/listings. Endpoint ini dirancang untuk mengambil SEMUA data di marketplace.
Database membaca 10.000+ data lelang dari hard-disk dan mengirimkannya ke Catalogue Service.
Catalogue Service bekerja keras mengalokasikan RAM (Heap Memory Java) untuk mengubah 10.000 data SQL tersebut menjadi objek JSON raksasa, lalu mengirimkannya lewat API Gateway ke Browser penjual.
Browser (Frontend) men- download JSON berukuran raksasa itu, lalu menjalankan fungsi JavaScript .filter() untuk membuang 9.990 data, hanya demi mendapatkan 10 data yang benar-benar milik penjual yang sedang login.
WARNING

Masalah Utama: Pendekatan ini termasuk buruk bagi skalabilitas (scalability). Sistem memboroskan memori database, menyiksa CPU backend untuk konversi JSON, memboroskan bandwidth jaringan (mengirim Megabytes data yang ujung-ujungnya dibuang), dan membuat browser pengguna menjadi lambat.

Paradigma Baru (Database Pushdown Filtering)
Kita melakukan optimasi dengan teknik Database Pushdown. Artinya, kita "mendorong" logika penyaringan sejauh mungkin ke hulu, yaitu langsung ke dalam mesin Database.

Saat halaman Seller Studio dibuka, ia memanggil endpoint khusus GET /api/v1/catalogue/listings/seller.
Catalogue Service tidak lagi meminta seluruh data. Ia menyuruh PostgreSQL untuk menyeleksinya langsung lewat query SQL yang presisi: SELECT * FROM listings WHERE seller_id = 'seller-1'.
Database Relasional (PostgreSQL) sangat ahli dan super kilat dalam melakukan seleksi baris. Database langsung mengeksekusi ini (menggunakan Index) dan HANYA mengembalikan 10 data ke Catalogue Service. 
Catalogue Service memproses 10 data tersebut dan membalas Frontend dengan payload JSON berukuran Kilobytes.

Rincian File yang Diubah:
Untuk mencapai optimasi di atas, saya telah memodifikasi 4 file di Backend dan 1 file di Frontend secara hati-hati tanpa merusak fitur utama:

Modifikasi Backend (bidmart-catalogue-service)

ListingRepository.java
Tindakan: Menambahkan method signature baru: List<Listing> findBySellerId(String sellerId);.
Fungsi: Mengizinkan Spring Data JPA untuk secara otomatis menciptakan fungsi pencarian SQL berbasis ID Penjual.

ListingService.java (Interface)
Tindakan: Menambahkan deklarasi method getListingsBySeller(String sellerId); agar kontrak service mendukung pengambilan data per-seller.

ListingServiceImpl.java (Implementasi)
Tindakan: Mengimplementasikan method getListingsBySeller untuk memanggil listingRepository.findBySellerId(sellerId).

ListingController.java
Tindakan: Menambahkan endpoint baru dengan anotasi @GetMapping("/seller").
Fungsi: Menyediakan titik masuk API yang membaca sellerId langsung dari HTTP Header X-User-Id.

Modifikasi Frontend (bidmart-frontend)

SellPage.tsx
Tindakan: Memodifikasi fungsi fetchSellerListings().
Fungsi: Mengubah URL dari /api/v1/catalogue/listings menjadi /api/v1/catalogue/listings/seller.
Mengganti fungsi fetch() standar dengan fungsi kustom authenticatedFetch() (agar token JWT secara otomatis disisipkan).
Menghapus logika .filter((listing) => String(listing.sellerId) === user.id) di dalam JavaScript, karena datanya sudah 100% tersaring bersih dari backend.
